# Multi-Problem Exchange with [NEXT_PROBLEM] -- Design

*Status: spec / 2026-06-09*

## Goal

Let the AI tutor walk through multiple teachable moments from the same real conversation in a single benchmark scenario. The tutor signals problem transitions with `[NEXT_PROBLEM]`; the system serves up the next problem from later in the original transcript. The full scenario ends when the tutor signals `[END]`, the queue of next problems empties, or `max_turns` is hit.

## Why

In the dyn_smoke run, scenarios stretched to 66-100 turns because the AI tutor kept inventing new problems on its own rather than handing off. `[END]` wasn't capturing the moment we cared about (the end of *one* teachable moment) -- it was conflated with "end of conversation." Splitting the signal makes the tutor's wrap-up behavior observable and lets the benchmark span multiple human-curated moments in sequence, more faithfully replaying the real conversation arc.

## Tokens (tutor system prompt v4)

Two end-of-turn signals, each on its own line at the very end of the tutor's reply:

- `[NEXT_PROBLEM]` -- "I'd hand off to the next problem now." The current moment is done; the system will introduce the next problem.
- `[END]` -- "I'm fully done with this student / conversation." Whole scenario stops.

`[NEXT_PROBLEM]` and `[END]` MAY appear together on the tutor's final message; `[END]` takes precedence (end immediately). If neither appears, the synthetic student goes next as today.

New prompt version: `prompts/benchmark/v4/tutor_system.txt` (= v3 verbatim minus the `[END]` paragraph, plus a new two-token instruction). v3 stays frozen. Default flips to `prompt_version: v4`. v4 students dir = byte-identical copy of v3 students.

## Pre-flight: per-scenario problem queue

When each scenario is materialized in `extract_human_scenarios`, we look up `ground_truth_hybrid/<conv_uuid>.json` and build an ordered queue of subsequent moments beyond the seed:

1. Filter to `situation_label_agg in {scaffolding, rigor}`.
2. Keep only moments whose `turn_start > seed.turn_end` (strictly after the seed moment, no overlap).
3. For each, collect cut votes and apply modal-cut selection + role adjustment (same logic the seed already uses). Drop moments where no valid cut survives.
4. Sort by `turn_start` ascending.

Stash the queue on the `Scenario` object as `next_problem_queue: list[{"turn_start", "turn_end", "cut_turn"}]`. Empty queue is fine -- the tutor just won't have anywhere to go on `[NEXT_PROBLEM]`.

## Exchange loop

Pseudocode (sync; batch follows the same control flow over `active_ids`):

```
ended = False
while not ended and len(generated_turns) < max_turns:
    tutor turn
    text, ends, next_prob = _parse_tutor_tokens(response.text)
    append tutor turn(s) (text only -- tokens stripped)

    if ends:
        ended = True
        ended_via = "END"
        break
    if next_prob:
        if scenario.next_problem_queue is empty:
            ended = True
            ended_via = "NEXT_PROBLEM_EXHAUSTED"
            break
        next_moment = scenario.next_problem_queue.pop(0)
        inject SYSTEM divider turn  # "--- New problem (source turns {ts}-{te}, cut {cut}) ---"
        inject the literal STUDENT text at next_moment.cut_turn from the ORIGINAL transcript
        record on exchange.problem_boundaries: {"after_turn": N, "source_moment": {...}}
        continue   # no synthetic student turn this round -- the injected turn IS the student

    if len(generated_turns) >= max_turns:
        ended_via = "MAX_TURNS"
        break

    student turn (synthetic, same persona across problems)

if not ended_via and len(generated_turns) >= max_turns:
    ended_via = "MAX_TURNS"
```

Notes:
- `[END]` takes precedence over `[NEXT_PROBLEM]` if both appear.
- Token stripping treats `[NEXT_PROBLEM]` like `[END]`: replace with empty, rstrip, return cleaned text.
- The SYSTEM divider is a real turn in `generated_turns` with `role="SYSTEM"`. It counts toward `max_turns`. Text format: `"--- New problem (source turns {turn_start}-{turn_end}, cut {cut_turn}) ---"`.
- The injected STUDENT turn uses the literal text from the real transcript at the popped moment's role-adjusted `cut_turn`. The synthetic student persona persists -- subsequent student turns are generated by the same student model with the running transcript (which now includes the injected divider + real-student text) as context.

## Helper additions in `benchmark/core/exchange.py`

- `NEXT_PROBLEM_TOKEN = "[NEXT_PROBLEM]"` constant.
- `_parse_tutor_tokens(text) -> (cleaned: str, ended: bool, next_problem: bool)` -- replaces `_check_end_token`. Detects both tokens; END wins on collision.
- `_inject_next_problem(...)` -- given the running exchange, the popped moment, and the original `conversation` dict, appends the SYSTEM divider + STUDENT turn(s) to `generated_turns` and the running transcript string. Returns updated `(running_transcript, next_turn_num)`.

## `Scenario` shape additions

- `next_problem_queue: list[dict]` -- ordered queue, possibly empty. Each entry has `{turn_start, turn_end, cut_turn}`. Populated by `extract_human_scenarios`.

`scenario_id` format unchanged. `cut_turn` stays the seed moment's role-adjusted cut.

## `Exchange` shape additions

- `problem_boundaries: list[dict]` -- e.g. `[{"after_turn": 12, "source_moment": {"turn_start": 45, "turn_end": 60, "cut_turn": 44}}]`. Empty by default.
- `ended_via: str` -- one of `"END"`, `"NEXT_PROBLEM_EXHAUSTED"`, `"MAX_TURNS"`. Set when `completed = True`.

Both default to safe values; older cached exchanges remain readable (`.get()` access).

## Role handling

`SYSTEM` becomes a third valid `role` value in `generated_turns`. Touched places:

- `_format_prefix` -- already iterates over all turns regardless of role, no change needed; SYSTEM lines render as `Turn N. SYSTEM: ...`.
- `_build_role_prompt` -- only builds prompts for TUTOR / STUDENT, no change.
- `annotator_bridge.build_synthetic_conversation` -- already passes turns through verbatim.
- Annotation pipeline -- SYSTEM turns appear inside the excerpt; the annotator just reads them as plain transcript context.
- Viewers (`view.py`, `view_replay.py`) -- add a styling class for `role=SYSTEM` (gray dividers). Cosmetic only.

## Default config

```yaml
benchmark:
  exchange:
    max_turns: 100
    prompt_version: v4    # was: v3
```

`max_turns` stays the absolute ceiling. We expect typical scenarios to walk 2-4 problems before [END] or exhaustion; if 100 turns is too generous in practice we can tune later.

## Tests (`tests/`)

Extend `tests/test_benchmark_exchange_dynamic.py`:

- `_parse_tutor_tokens`: detects END alone, NEXT_PROBLEM alone, both (END wins), neither.
- `run_exchange` with empty queue: tutor emits `[NEXT_PROBLEM]` -> ended_via=`"NEXT_PROBLEM_EXHAUSTED"`, no injection.
- `run_exchange` with one queued problem: tutor emits `[NEXT_PROBLEM]` -> SYSTEM divider + injected STUDENT turn appear in `generated_turns`; `problem_boundaries` records the transition; tutor turn after injection is a fresh model call.
- `run_exchange` with two queued problems: tutor emits `[NEXT_PROBLEM]` twice -> two injections; second `[NEXT_PROBLEM]` after queue empties -> ended_via=`"NEXT_PROBLEM_EXHAUSTED"`.
- `run_exchange` with `[END]` precedence: text contains both tokens, scenario ends, no injection.
- `run_exchanges_batch`: per-scenario queue handling stays independent across scenarios.

Extend `tests/test_benchmark_human_scenarios.py`:

- `next_problem_queue` is populated with subsequent moments (filtered to scaff/rigor + cut_turn, sorted by turn_start, modal-cut + role-adjusted).
- Queue is empty when the seed is the last moment in the conversation.
- A moment with `turn_start <= seed.turn_end` is excluded from the queue.

## What's NOT changing

- Single-moment scenarios still run -- they just have an empty queue and end on `[END]` / `max_turns`.
- Modal-cut selection logic unchanged.
- Annotator pipeline (v12 prompts, suggestion text) unchanged.
- Phase 2 / Phase 3 / scoring unchanged.

## Out of scope

- Cross-conversation problem banks.
- Adaptive "should we move on?" judgement (relying entirely on tutor's own signal).
- Re-annotating per-problem segments separately (annotator still treats each scenario as one transcript; downstream analysis can use `problem_boundaries` to split if needed).

## Risk / open questions

- **Tutors may overuse `[NEXT_PROBLEM]`** and skim through problems without genuinely teaching. Mitigation: the prompt explicitly says "only when the student has reached an answer they can run with, the misconception is resolved, or the problem is finished." If observed in practice, tighten language or count NEXT_PROBLEM emissions in scoring.
- **Injected real-student turn may reference prior context the AI tutor doesn't have** (e.g., "and the answer to my last question was 12, but on this one..."). The AI tutor will just have to handle the ambiguity; downstream we'll see if this produces noisy continuations.
- **SYSTEM role addition** -- callers that strictly enumerate `{TUTOR, STUDENT}` may need a defensive default. Search and patch (annotator excerpt builder is the main risk).
- **Queue construction cost** -- per scenario we load the GT file again. For 1k scenarios this is cheap; if it grows, cache GT by conv_id in `extract_human_scenarios`.
